BAGIAN I: KONSEP DASAR & VISI
=================================================================

1. VISI DAN TUJUAN APLIKASI
   Membangun aplikasi simpan pinjam yang mampu mengelola:
   - Hierarki organisasi: BOS → UNIT → CABANG
   - Produk simpanan berkala (harian/mingguan) dengan imbal hasil "kewer/mawar"
   - Pinjaman dengan angsuran harian/mingguan untuk pedagang pasar & UMKM
   - Transaksi kas frekuensi tinggi dengan dukungan petugas lapangan (kolektor)
   - Deteksi risiko kredit melalui hubungan keluarga
   - Pencegahan fraud internal (kutipan tidak disetor, pelarian dana)
   - Laporan real-time dan dashboard analitik

BAGIAN II: ARSITEKTUR SISTEM
=================================================================

2. ARSITEKTUR TEKNOLOGI

   2.1 Stack Teknologi
       - Frontend: HTML, Bootstrap, jQuery, AJAX
       - Backend: PHP native dengan REST API
       - Database: MySQL (satu instance dengan beberapa schema logis)
       - Pattern: Hybrid (bukan SPA murni)
       - Halaman utama (index.php) memuat konten dinamis via AJAX

   2.2 Struktur Folder
       /api/          - Endpoint REST (auth, members, savings, loans, dll)
       /pages/        - File HTML untuk setiap menu (dashboard, dll)
       /assets/       - CSS, JS, vendor libraries
       /config/       - Koneksi database, fungsi helper

3. ARSITEKTUR DATABASE: SATU INSTANCE, MULTI-SCHEMA

   Menggunakan satu database server dengan tiga schema logis:

   3.1 schema_person
       - Data personal anggota
       - Hubungan keluarga (family_links)
       - Data biometrik

   3.2 schema_address
       - Data alamat (provinsi, kabupaten, kecamatan, desa)
       - Detail alamat dengan tipe: rumah, kantor, penagihan

   3.3 schema_app
       - Data operasional (unit, cabang, anggota)
       - Simpanan, pinjaman, transaksi
       - Kas, audit log, notifikasi

   3.4 Keuntungan Arsitektur Ini
       - JOIN lintas schema tetap dapat dilakukan (prefix schema)
       - Foreign key dapat dibuat antar schema (InnoDB support)
       - Backup dan maintenance lebih sederhana
       - Skalabilitas dengan sharding jika diperlukan

BAGIAN V: FITUR LANJUTAN

9. MANAJEMEN LIMIT & PLAFON PINJAMAN

   9.1 Konsep
       - Setiap user dengan otoritas approval memiliki limit plafon
       - Dikonfigurasi terpusat oleh BOS
       - Approval otomatis jika dalam limit
       - Escalation ke atasan jika melebihi limit

   9.2 Implementasi
       - Tambah kolom approval_limit pada tabel users
       - Saat pengajuan pinjaman, sistem cek limit
       - Jika <= limit: status "approved" (jika analisa OK)
       - Jika > limit: status "proposed", perlu approval atasan
       - Audit log mencatat semua persetujuan

   9.3 Contoh SQL
       ALTER TABLE users ADD COLUMN approval_limit DECIMAL(15,2) DEFAULT 0;

10. PERHITUNGAN BUNGA & DENDA

    10.1 Metode Bunga
         - Flat: Bunga tetap dari pokok pinjaman
         - Efektif: Bunga dari sisa pokok (menurun)
         - Anuitas: Angsuran tetap, komposisi berubah
         - Pilihan metode per produk pinjaman

    10.2 Perhitungan Angsuran
         Berdasarkan:
         - Jumlah pinjaman
         - Suku bunga per tahun
         - Frekuensi (harian/mingguan/bulanan)
         - Jangka waktu (jumlah periode)
         - Hasil disimpan di tabel loan_installments

    10.3 Denda Keterlambatan
         - Denda per hari setelah due_date
         - Persentase denda per produk (misal 0,1%/hari)
         - Cron job malam hari cek overdue
         - Insert ke tabel penalties
         - Dapat dibayar bersamaan angsuran berikutnya

    10.4 Contoh Cron Implementation
         UPDATE loan_installments SET status = 'late'
         WHERE due_date < CURDATE() AND status = 'pending';
         
         INSERT INTO penalties (member_id, reference_type, reference_id, amount)
         SELECT l.member_id, 'loan_installment', li.id,
                (li.amount * penalty_rate / 100)
         FROM loan_installments li
         JOIN loans l ON li.loan_id = l.id
         WHERE li.status = 'late' AND li.penalty_added = 0;

11. PENANGANAN TRANSAKSI OFFLINE (MOBILE COLLECTOR)

    11.1 Arsitektur Offline
         - PWA (Progressive Web App) dengan service worker
         - Data transaksi di localStorage atau IndexedDB
         - ID unik lokal (UUID) untuk setiap transaksi
         - Cache API untuk data master (anggota, produk)

    11.2 Sinkronisasi
         - Batch upload saat koneksi tersedia
         - Endpoint: POST /api/collector/sync
         - Server validasi setiap transaksi
         - Response: daftar sukses dan gagal
         - Hapus data lokal yang sudah berhasil

    11.3 Penanganan Konflik
         - Cek duplikasi di server
         - Tolak jika sudah dicatat kasir
         - Kartu/kode unik untuk identitas anggota
         - GPS tracking untuk audit
         - Timestamp untuk resolusi konflik

12. BACKUP & DISASTER RECOVERY

    12.1 Backup Otomatis
         - Full backup harian (pukul 02:00)
         - Incremental backup setiap 6 jam
         - Lokasi: Cloud storage (AWS S3/GCS) + server sekunder
         - Retention: harian 30 hari, bulanan 1 tahun

    12.2 Prosedur Pemulihan
         - Dokumen prosedur tersedia dan diuji berkala
         - Failover ke replica < 1 jam
         - Restore backup ke server baru untuk bencana besar
         - Konfigurasi server (ansible/docker) untuk deployment cepat

    12.3 Redundansi Infrastruktur
         - VPS dengan snapshot otomatis
         - Database replication (master-slave)
         - Load balancer untuk high availability (opsional)

    12.4 Pengujian Restore
         - Uji restore bulanan ke staging
         - Validasi integritas data
         - Dokumentasi hasil testing

24. MACHINE LEARNING UNTUK APLIKASI SIMPAN PINJAM

    Machine learning diintegrasikan sebagai microservice terpisah (Python API)
    yang dipanggil oleh aplikasi utama.

    24.1 Prediksi Risiko Kredit (Credit Scoring)
         - Model: Logistic Regression, Random Forest, XGBoost
         - Data Input:
           * Riwayat pinjaman dan pembayaran
           * Jumlah pinjaman dan pendapatan
           * Riwayat simpanan
           * Hubungan keluarga
           * Data demografi
         - Output: Skor risiko (0-100), rekomendasi plafon
         - Implementasi: API ML dipanggil saat pengajuan pinjaman
         - Hasil ditampilkan ke analis sebagai pertimbangan

    24.2 Deteksi Dini Potensi Default
         - Model: Survival analysis, time-series classification
         - Data Input:
           * Pola pembayaran angsuran
           * Frekuensi keterlambatan
           * Perubahan perilaku transaksi
         - Output: Peringatan dini untuk anggota berisiko
         - Implementasi: Cron job mingguan prediksi semua pinjaman aktif
         - Alert ke kolektor/kepala cabang

    24.3 Deteksi Anomali untuk Fraud Internal & Eksternal
         - Model: Isolation Forest, Autoencoder
         - Data Input:
           * Log transaksi kasir dan kolektor
           * Waktu, lokasi (GPS), nominal transaksi
           * Pola setoran
         - Output: Deteksi transaksi tidak biasa
         - Implementasi: Monitoring real-time
         - Alert otomatis jika anomali terdeteksi

    24.4 Rekomendasi Produk Simpanan/Pinjaman
         - Model: Collaborative filtering, content-based filtering
         - Data Input:
           * Profil anggota (usia, pekerjaan)
           * Riwayat produk
           * Perilaku transaksi
         - Output: Rekomendasi produk yang sesuai
         - Implementasi: Dashboard anggota, tools untuk kolektor

    24.5 Clustering Anggota untuk Segmentasi
         - Model: K-Means clustering
         - Data Input:
           * Karakteristik anggota
           * Pendapatan, frekuensi transaksi
           * Plafon pinjaman
         - Output: Segmen anggota (pedagang pasar, buruh harian, dll)
         - Implementasi: Strategi penagihan dan penawaran produk

    24.6 Optimalisasi Rute Kolektor
         - Model: TSP algorithm dengan reinforcement learning
         - Data Input: Lokasi anggota dengan tagihan hari ini
         - Output: Rute optimal untuk kolektor
         - Implementasi: Dikirim ke mobile app setiap pagi

    24.7 Implementasi Teknis ML
         - Infrastruktur: Microservice Python (Flask/FastAPI)
         - REST endpoint untuk setiap model
         - Training: Data historis minimal 6 bulan
         - Retraining: Bulanan atau saat ada data baru signifikan
         - Keamanan: Hashing/tokenization untuk data sensitif
         - Monitoring: Track accuracy dan performance model

    24.8 Keuntungan Machine Learning
         - Mengurangi risiko kredit macet 20-30% (estimasi)
         - Deteksi fraud lebih awal
         - Personalisasi layanan meningkatkan loyalitas
         - Efisiensi operasional kolektor
         - Data-driven decision making
